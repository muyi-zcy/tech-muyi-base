# 02 基础能力模块手册

## 1. 模块说明

基础能力模块是所有业务服务的底座，建议作为默认依赖：

- `tech-muyi-base-core`
- `tech-muyi-base-common`
- `tech-muyi-base-config`
- `tech-muyi-base-util`
- `tech-muyi-base-exception`
- `tech-muyi-base-id`
- `tech-muyi-base-on`

## 2. core 模块

### 2.1 主要职责

- 提供 Web、线程池、数据库分页、序列化等核心配置能力。
- 提供统一异常处理器和基础 DAO/Service 查询能力。

### 2.2 常见能力点

- `GlobalExceptionHandle`：全局异常兜底。
- `AsyncConfiguration`、`MyTaskDecorator`：异步任务上下文传递。
- `PageConfig`：MyBatis Plus 分页能力。
- `JacksonConfig`、`EnumDeserializer`：统一 JSON 反序列化策略。

### 2.3 核心方法与用法（重点）

#### `GlobalExceptionHandle`

- `handleMyException(MyException ex)`：已知业务异常统一转换为 `MyResult.fail(ex)`。
- `handleThrowable(Throwable t)`：未知异常包装为 `UnknownException`，避免裸异常外泄。

#### `MyTaskDecorator`

- `decorate(Runnable runnable)`：异步任务执行前绑定请求上下文与 TTL 上下文，执行后清理上下文，避免线程池串数据。

#### 使用示例（异步上下文保留）

```java
@Bean("bizExecutor")
public ThreadPoolTaskExecutor bizExecutor() {
    ThreadPoolTaskExecutor executor = new ThreadPoolTaskExecutor();
    executor.setCorePoolSize(8);
    executor.setMaxPoolSize(16);
    executor.setTaskDecorator(new MyTaskDecorator());
    executor.initialize();
    return executor;
}
```

### 2.4 使用建议

- 业务代码不要重复实现全局异常处理；扩展时优先复用统一返回模型。
- 异步线程池需结合上下文透传能力，避免 traceId 丢失。

## 3. common 模块

### 3.1 主要职责

- 统一业务返回对象、公共枚举、通用查询参数对象。

### 3.2 核心对象建议

- `MyResult`：统一成功/失败响应结构，接口层保持单一输出标准。
- `CommonEnum`、`ConditionEnum`：控制公共语义，避免字符串硬编码。
- `MyBaseQuery`：封装分页与通用筛选参数。

### 3.3 `MyResult` 核心方法

- `MyResult.ok()`：成功无数据。
- `MyResult.ok(T data)`：成功返回数据。
- `MyResult.ok(T data, MyBaseQuery query)`：成功返回数据与分页元信息。
- `MyResult.fail(String errCode, String errMsg)`：按错误码返回失败。
- `MyResult.fail(Throwable throwable)`：自动识别 `MyException`，否则降级为未知异常。

## 4. config 模块

### 4.1 主要职责

- 管理系统级常量、开关、环境初始化流程。

### 4.2 配置实践

- 常量集中在 `Constants`，防止各模块重复声明。
- 初始化逻辑收敛到 `InitConfiguration`，便于启动排错。
- 系统配置统一映射到 `MyProperties`，避免散落 `@Value`。

## 5. util 模块

### 5.1 主要职责

- 提供 JSON、日期、IP、请求、对象映射、线程上下文等通用工具。

### 5.2 使用建议

- JSON 序列化优先使用统一工具，避免多个 ObjectMapper 导致结果不一致。
- 请求上下文获取统一走工具类，降低框架升级影响面。
- TTL 相关工具仅用于需要跨线程透传的数据，避免滥用导致内存泄漏。

### 5.3 `MyTtlContextManager` 核心方法

- `register(MyTransmittableThreadLocal ttl)`：注册可透传上下文变量。
- `downAllData()`：把当前线程上下文转成 `Map<String, String>` 用于下传（如 RPC attachments）。
- `upAllData(Map<String, String>)`：把下传数据恢复到当前线程。

示例：

```java
MyTtlContextManager.register(REQUEST_ID_TTL);
REQUEST_ID_TTL.set("req-123");
Map<String, String> data = MyTtlContextManager.downAllData();
MyTtlContextManager.upAllData(data);
```

## 6. exception 模块

### 6.1 主要职责

- 定义统一异常体系、错误码接口、错误码扫描与暴露能力。

### 6.2 异常模型

- `MyException`：业务异常基类。
- `UnknownException`：未知异常兜底。
- `CommonErrorCodeEnum`：系统级错误码。
- `ErrorCodeHelper`：聚合和查询全量错误码。

### 6.3 `ErrorCodeHelper` 核心方法

- `getAllErrorCode()`：扫描实现 `BaseErrorInfoInterface` 的枚举，并按 `@ErrorCodeInfoAnno` 聚合为树状错误码列表。

### 6.4 错误码实践

- 按模块分配错误码区间，避免跨服务冲突。
- 一个错误码对应唯一语义，禁止复用同码多义。
- 错误描述面向排障可读，避免模糊文案。

## 7. id 模块

### 7.1 主要职责

- 提供分布式 ID 生成能力（雪花算法相关实现）。

### 7.2 关键组件

- `MyIdGenerator`：统一 ID 生成入口。
- `MyIdGeneratorConfig`：生成器装配。
- `DefaultSnowflakeImpl`：雪花实现。

### 7.3 接入建议

- 数据库主键、自定义业务单号可统一使用该模块。
- 多机部署时需确保 workerId/datacenterId 配置无冲突。
- 详细配置与示例见 `doc/10-id模块使用说明.md`。

## 8. on 模块

### 8.1 主要职责

- 作为基础依赖聚合模块，简化业务服务引入成本。

### 8.2 使用建议

- 新业务服务优先依赖 `on` 模块快速接入，再按需叠加能力模块。

## 9. 典型落地模板

1. Controller 层统一返回 `MyResult`。
2. Service 层抛出 `MyException`（带明确错误码）。
3. DAO 层使用 core 提供的查询与分页能力。
4. 异步任务必须经统一线程池与上下文装饰器执行。
