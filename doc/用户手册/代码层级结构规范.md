# 代码层级结构规范

本文档用于约束基于生成器产出的工程代码，目标是：**职责单一、边界清晰、便于维护、避免耦合失控**。

## 1. 分层模型（强制）

标准分层如下：

- `web`：Controller（接口入参与协议层）
- `client`：`dto/query/api`（跨模块调用契约）
- `core`：`service/manager/dao/entity`（领域与数据访问）
- `mapper`：MyBatis 映射文件（仅由 DAO 使用）

## 2. 调用方向（强制）

只允许单向依赖：

- `web/controller` -> `core/service` 或 `client/api`
- `core/service` -> `core/manager` -> `core/dao`
- `core/dao` -> `mapper`

禁止反向和跨层绕过。

## 3. 同层调用规则（强制）

- **禁止同层互调**：`ServiceA` 不得直接调用 `ServiceB`。
- 如需复用，抽到 `Manager`（领域编排）或公共组件。
- `Controller` 之间禁止互调。
- `Dao` 之间禁止互调。

## 4. 跨表规则（强制）

- 一个 `Dao` 只服务一个聚合根（通常一张主表及其固定附属表）。
- **禁止在业务代码中随意跨表拼装查询**。
- 需要跨表场景时，必须：
  1. 先定义明确业务聚合边界；
  2. 在 `Manager` 层编排多个 `Service/Dao` 调用；
  3. 保持每个 DAO 的单表职责不变。

## 5. SQL 规则（强制）

- **禁止在 Java 代码里写 SQL 字符串**（包括 `Service/Manager/Controller`）。
- SQL 仅允许出现在 Mapper XML（或统一 DAO 抽象能力）中。
- **禁止在 Controller、Service、Manager 中使用 JDBC 直连**。

## 6. 各层职责约束（强制）

### 6.1 Controller 层

- 只做协议转换、参数基础校验、调用编排入口。
- 返回统一 `MyResult`。
- 禁止写业务规则、禁止访问 DAO。

### 6.2 Service 层

- 实现单一业务用例。
- 可调用 `Manager` 和本域 `DAO`（按项目约定择一）。
- 禁止直接处理 HTTP 对象，禁止拼 SQL。

### 6.3 Manager 层

- 负责跨领域编排、事务边界协同、复杂规则聚合。
- 禁止感知 Web 协议对象。
- 禁止越过 DAO 直接操作数据库连接。

### 6.4 DAO 层

- 负责持久化访问。
- 入参出参以实体/查询对象为主，不泄露上层协议模型。
- 禁止包含业务流程判断。

### 6.5 DTO/Query/API（client 层）

- `Query`：查询入参模型。
- `Dto`：跨层/跨服务输出模型。
- `ApiService`：对外契约接口，不承载实现逻辑。

## 7. 事务与一致性规则

- 事务边界默认放在 `Service` 或 `Manager`，不得下沉到 Controller。
- 多 DAO 操作必须在同一业务边界内显式事务控制。
- 跨服务一致性通过事件/补偿实现，禁止分布式事务滥用。

## 8. 异常与返回规范

- 业务异常统一抛 `MyException`（带错误码）。
- 未知异常统一由全局异常处理器接管。
- Controller 层禁止 `try-catch` 后吞异常。

## 9. 命名与文件规范

- 一个类只表达一个职责。
- 类名与表名/业务语义一致（生成器默认驼峰转换后可手工修正）。
- Mapper 文件名与 DAO/实体保持可追踪关系。

## 10. 禁止项清单（可直接贴代码评审）

- 禁止 `Controller -> DAO`。
- 禁止 `ServiceA -> ServiceB` 同层互调。
- 禁止 `DAOA -> DAOB` 同层互调。
- 禁止 Java 字符串 SQL。
- 禁止非 DAO 层使用 `JdbcTemplate/Connection/Statement`。
- 禁止跨表逻辑散落在多个 Service 中。
- 禁止在接口层做复杂业务判断。

## 11. 推荐落地机制

- 在代码评审模板中加入“分层/跨表/SQL 禁止项”检查项。
- 增加静态扫描规则（如 ArchUnit/自定义 Checkstyle）校验包依赖方向。
- PR 必须说明：是否新增跨聚合调用、是否新增 SQL、事务边界放在哪一层。

## 12. 例外申请机制

出现必须破例场景（历史包袱/性能紧急优化）时，必须满足：

1. 在 PR 描述中写明破例点与原因；
2. 给出回收计划（何时恢复标准分层）；
3. 通过架构评审后方可合并。
