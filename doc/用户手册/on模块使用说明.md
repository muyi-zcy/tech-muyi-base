# on 模块使用说明

## 1. 模块定位

`tech-muyi-base-on` 是基础能力聚合模块，用于简化新服务接入成本。  
它通过一次依赖引入常用基础模块，适合“先快速启动，再按需细化”的研发流程。

## 2. 适用场景

- 新服务初始化阶段，希望尽快具备统一基础能力；
- 业务团队希望降低依赖装配复杂度；
- 模块拆分策略尚未完全确定，先以聚合依赖落地。

## 3. 聚合策略

`on` 模块通常聚合如下基础能力：

- `core`
- `common`
- `config`
- `util`
- `exception`
- `id`（按项目策略可选）

建议以实际 `pom` 为准进行确认。

## 4. 依赖接入

```xml
<dependency>
    <groupId>tech.muyi</groupId>
    <artifactId>tech-muyi-base-on</artifactId>
</dependency>
```

## 5. 使用建议

- **先聚合后拆分**：项目初期优先使用 `on`，业务稳定后再按需替换为细粒度模块依赖。
- **关注传递依赖**：排查版本冲突时，优先检查 `on` 引入的传递依赖链。
- **统一版本管理**：建议在父 `pom` 统一管理基础模块版本，避免子模块漂移。

## 6. 迁移建议（从 on 到细粒度依赖）

当服务进入稳定期、对依赖有更强控制诉求时，可逐步迁移：

1. 识别当前服务真实使用的基础能力；
2. 显式引入对应模块（如 `core`、`common`、`exception`）；
3. 移除 `on` 并进行编译、测试、启动验证；
4. 在团队模板中固化新的依赖基线。

## 7. 常见问题

### 7.1 引入 on 后包体积变大

这是聚合依赖的典型现象。  
如果服务对体积敏感，可改为按需引入独立模块。

### 7.2 不确定某能力来自哪个模块

建议先从 `doc/用户手册/基础能力模块手册.md` 查看模块边界，再定位到对应独立文档深入阅读。
