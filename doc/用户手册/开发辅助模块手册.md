# 开发辅助模块手册

## 1. 模块范围

- `tech-muyi-base-api`
- `tech-muyi-base-generator`
- `tech-muyi-base-dmask`
- `tech-muyi-base-boot`
- `tech-muyi-base-dependencies`

## 2. API 文档模块（tech-muyi-base-api）

### 2.1 功能定位

- 基于 Swagger/Knife4j 输出接口文档和在线调试能力。

### 2.2 关键组件

- `Swagger3Config`：文档扫描与元信息配置。
- `ApiConfigProperties`：文档配置项映射。

### 2.3 核心方法与配置（重点）

#### `Swagger3Config#createRestApi()`

- local/dev 环境启用文档，其它环境默认关闭。
- 统一注册 GET/POST 全局响应（如 404）。
- 扫描 Controller 并生成 OpenAPI 文档。

#### `Swagger3Config#apiInfo()`

- 从 `ApiConfigProperties` 读取标题、描述、版本、联系人。
- 配置缺失时使用默认值兜底，避免文档初始化失败。

#### `ApiConfigProperties`

- 前缀：`muyi.api`
- 字段：`title`、`desc`、`version`、`contact.name/url/email`

配置示例：

```yaml
muyi:
  api:
    title: 用户中心接口文档
    desc: 用户中心服务 API
    version: 1.2.0
    contact:
      name: platform-team
      url: https://github.com/muyi-zcy
      email: dev@example.com
```

### 2.4 使用建议

- 生产环境按策略关闭调试入口或启用鉴权。
- Controller 注解描述保持完整，减少前后端沟通成本。
- 版本化 API 时在分组中体现版本维度。

## 3. 代码生成模块（tech-muyi-base-generator）

### 3.1 功能定位

- 提供 DAO、Service、Controller、DTO、Query 等模板化代码生成。

### 3.2 关键能力

- 一次性生成标准分层代码骨架。
- 减少重复 CRUD 代码和命名不一致问题。

### 3.3 核心方法与流程（重点）

`MyGeneratorCode` 的主流程：

- `main()`：启动 Swing 界面。
- `testConnection(...)`：测试数据库连接。
- `queryTables(...)`：读取库表列表（`SHOW TABLES`）。
- `buildConfig(...)`：收集 UI 参数并构建生成配置。
- `detectExistingFiles(...)`：预扫描即将生成的同名文件。
- `generateForTables(...)`：按选中模板调用 `Controller/Query/Dto/.../Manager.deal(...)` 批量生成。
- `saveConfig(...)` / `loadConfig(...)`：持久化本地配置到 `~/.tech-muyi-generator-config.json`。

建议操作顺序：

1. 填好数据库连接与 `groupId`。
2. 点击“加载表”并选表。
3. 选择模板集合。
4. 若要覆盖旧文件，显式勾选“覆盖已存在文件”。
5. 先生成到测试目录验证，再导入正式工程。

### 3.4 使用建议

- 生成代码后务必做业务语义补全，不直接上线模板代码。
- 将生成器模板纳入团队规范，保持风格统一。

## 4. 数据脱敏模块（tech-muyi-base-dmask）

### 4.1 功能定位

- 统一处理日志、接口返回、持久化过程中的敏感信息掩码。

### 4.2 使用建议

- 明确敏感字段清单（手机号、证件号、银行卡、地址、邮箱）。
- 在日志链路和接口层双重脱敏，避免遗漏。

### 4.3 落地建议（当前仓库）

当前模块源码为空，建议后续补充以下基础能力并在本手册固化：

- 注解式脱敏（如 `@Mask(type = PHONE)`）。
- Jackson 序列化层自动脱敏（响应与日志分层控制）。
- 日志切面脱敏器（与 `RestApiFilter` 联动）。

## 5. Boot 与 Dependencies 模块

### 5.1 功能定位

- `tech-muyi-base-boot`：单体脚手架聚合入口，默认模块更偏本地能力（core/api/log/config/util/id）。
- `tech-muyi-base-dependencies`：Dubbo 集群脚手架聚合入口，默认包含 RPC 与 Redis（core/rpc/api/redis/util/id）。

### 5.2 依赖差异（按当前 POM）

`tech-muyi-base-boot` 包含：

- `tech-muyi-base-core`
- `tech-muyi-base-api`
- `tech-muyi-base-log`
- `tech-muyi-base-config`
- `tech-muyi-base-util`
- `tech-muyi-base-id`

`tech-muyi-base-dependencies` 包含：

- `tech-muyi-base-core`
- `tech-muyi-base-rpc`
- `tech-muyi-base-api`
- `tech-muyi-base-redis`
- `tech-muyi-base-util`
- `tech-muyi-base-id`

### 5.3 选型规则

- 纯单体服务优先 `tech-muyi-base-boot`。
- 需要 Dubbo 服务注册/调用优先 `tech-muyi-base-dependencies`。
- 不建议两者同时引入，避免重复依赖与自动配置冲突。

### 5.4 使用建议

- 业务项目尽量继承统一聚合入口，不在子模块散落版本号。
- 新增三方库先在聚合入口统一纳管后再推广到业务服务。

## 6. 团队提效实践建议

1. 建立模块接入模板仓库，减少新服务初始化成本。
2. 将生成器输出与代码规范检查绑定，自动拦截低质量提交。
3. 文档模块化维护，版本发布时同步更新变更记录。
