# 05 任务与调度模块手册

## 1. 模块范围

- `tech-muyi-base-easyjob`
- `tech-muyi-base-elasticjob`

## 2. EasyJob（轻量定时任务）

### 2.1 功能定位

- 适用于单体或轻量服务内的本地定时任务。

### 2.2 适用场景

- 数据清理、状态轮询、缓存预热等低复杂度任务。

### 2.3 核心类与方法（重点）

#### `BaseTask`

- 核心方法：`run()`。
- 所有 EasyJob 任务都实现该接口，业务逻辑写在 `run()` 中。

#### `@MyEasyJob`

- `id`：任务唯一标识（建议全局唯一）。
- `cron`：cron 表达式。
- `jobName`：任务展示名，主要用于日志。

#### `MyEasyJobAutoConfiguration`

- `configureTasks(ScheduledTaskRegistrar)`：扫描容器里的 `BaseTask` Bean 并注册触发器。
- `buildTask(...)`：按是否存在 `RedissonManage` 决定单机执行还是分布式互斥执行。
- `process(BaseTask)`：单机执行流程。
- `lockProcess(BaseTask)`：集群执行流程（Redis 锁 + `:once` 执行时间幂等校验）。
- `getThisTaskTime(String cron)`：计算当前周期对应时间点，用于跨节点“本次任务是否已执行”判断。

### 2.4 快速示例

```java
@Component
@MyEasyJob(id = "demo:sync", jobName = "同步演示任务", cron = "0 */5 * * * ?")
public class DemoSyncTask implements BaseTask {
    @Override
    public void run() {
        // 执行业务逻辑
    }
}
```

### 2.5 使用建议

- 任务逻辑保持幂等，防止重复触发导致数据异常。
- 单次任务耗时不可过长，超长任务建议拆分或异步化。
- 对失败任务增加可观测指标和告警。

## 3. ElasticJob（分布式任务）

### 3.1 功能定位

- 面向多实例部署场景，提供分片执行和故障转移能力（依赖 Zookeeper）。

### 3.2 适用场景

- 大规模数据处理、分片批处理、跨节点任务并行执行。

### 3.3 核心类与方法（重点）

#### `@MyElasticJob`

- `id`：任务唯一 ID。
- `cron`：执行周期。
- `jobName`：任务名（会用于 `JobCoreConfiguration`）。
- `shardingTotalCount`：分片总数。
- `shardingItemParameters`：分片参数，如 `0=A,1=B`。
- `jobParameter`：任务参数（字符串）。
- `dataSource`：事件追踪库 Bean 名（可选）。
- `disabled`：是否禁用，默认 `true`。
- `overwrite`：是否覆盖注册中心配置。

#### `MyElasticJobAutoConfiguration`

- `initElasticJob()`：读取 ZK 配置并初始化 `ZookeeperRegistryCenter`。
- `initSimpleJob(...)`：扫描 `SimpleJob` + `@MyElasticJob` 并组装配置。
- `bulidElasticJob(...)`：根据是否有 `dataSource` 决定是否启用 `JobEventRdbConfiguration`。

### 3.4 快速示例

```java
@Component
@MyElasticJob(
        id = "order:reconcile",
        jobName = "订单对账",
        cron = "0 0/10 * * * ?",
        shardingTotalCount = 2,
        shardingItemParameters = "0=beijing,1=shanghai",
        overwrite = true,
        disabled = false
)
public class OrderReconcileJob implements SimpleJob {
    @Override
    public void execute(ShardingContext shardingContext) {
        // 根据分片参数执行业务
    }
}
```

### 3.5 配置与部署建议

- 统一管理作业命名规则，避免重复注册冲突。
- 分片总数依据 CPU 与业务数据规模动态评估。
- 生产环境开启作业监控和执行日志追踪。

### 3.6 任务设计原则

- 幂等：任何分片可重复执行。
- 可中断：任务支持暂停、恢复、补偿。
- 可观测：记录开始时间、结束时间、处理量、失败原因。

## 4. 任务模块选型建议

- 仅单节点或低并发场景：优先 `easyjob`。
- 多节点高并发与可扩展场景：优先 `elasticjob`。
- 同时存在两类任务时，建议按业务边界隔离，不混合调度框架职责。

## 5. 任务治理清单

- 是否设置任务超时时间。
- 是否有失败重试与补偿策略。
- 是否建立任务级监控指标（成功率、耗时、积压量）。
- 是否建立统一告警接入（短信、邮件、IM）。
